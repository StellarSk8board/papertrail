## First screenshot / social preview strategy

### Preferred order
1. **Social preview image:** clean static image, not the GIF
2. **README hero image:** keep the demo GIF lower on the page
3. **Release page screenshots:** use 1–3 stills that show the office and terminal/chat clearly

### Best social preview candidate
Use `build/screenshot.png` as the starting point if it is crisp and readable at GitHub card size.

### What the social preview should communicate
At a glance, it should show:
- this is a desktop app
- this is agent orchestration, not a chatbot skin
- it has a distinctive pixel-office identity
- it is actively being ported to Windows

### Recommended overlay text
Keep it short:
- `PaperTrail`
- `Claude-powered AI employees`
- `Windows beta port`

Do not cram feature bullets into the social preview. The card only gets one job.

## README top-of-page strategy

The README should do this in order:
1. Say what the app is
2. Say platform status honestly
3. Show the product visually
4. Explain why someone should care
5. Tell developers how to run it

If the README starts sounding like a launch page written by a committee, cut it.
